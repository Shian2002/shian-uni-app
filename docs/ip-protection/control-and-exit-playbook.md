# 控制权与退出预案

本文档用于防止项目进入公司运营后，个人被移出协作而失去对软件成果、证据链和后续谈判筹码的控制。

## 基本判断

完整源码一旦交付给公司，技术上无法阻止复制、学习或再次部署。因此不要依赖隐藏逻辑、后门、停服开关或破坏性限制来保护权益。这类做法会损害信任，也可能在争议中反过来对个人不利。

更稳的保护方式是：

1. 保留原始仓库、域名、证据包和软著/商标申请资料。
2. 公司获得运营使用权，而不是当然取得所有权。
3. 只给公司完成运营所需的最小权限。
4. 每次主体、部署、支付或仓库权限变更前，先导出证据快照。
5. 合作终止时，个人仍能证明早期成果、继续开发，并主张授权费、分成或停止使用。

## 不建议交出去的控制权

- 原始 Git 仓库 owner。
- 域名注册账号和域名所有权。
- 软著账号、软著申请材料原件和验证码。
- 商标申请账号。
- 个人证据包、完整 Git bundle、私有说明。
- 服务器 root 长期唯一控制权。
- 生产数据库原始备份。
- 支付证书、API key、商户密钥的个人留存副本。

## 可以给公司的权限

- 公司备案所需的网站主办资料。
- 公司微信支付商户配置接入权限。
- 项目部署所需的受限服务器账号。
- 公司运营后台账号。
- 公司客服、开票、用户协议和隐私政策主体信息。
- 一个镜像仓库或协作仓库的开发权限。

## 仓库交接建议

优先方案：

- 个人保留原始仓库 owner。
- 公司以 collaborator 或团队权限参与。
- 公司需要内部沉淀时，可以建立公司镜像仓库，但不要删除个人原始仓库。
- 公司仓库中保留 `COPYRIGHT.md`、`AUTHORS.md`、`NOTICE.md` 和 `docs/ip-protection/` 相关台账。

如果公司强烈要求迁移到公司仓库：

1. 迁移前导出完整 Git bundle。
2. 迁移前导出源码证据包和哈希。
3. 迁移前保留远程仓库页面截图、提交历史截图、成员权限截图。
4. 迁移说明中写明“迁移用于协作和运营，不代表知识产权转让”。
5. 个人账号保留管理员或至少可读权限。

## 部署和服务器建议

- 生产部署账号与个人开发账号分开。
- 服务器 root 权限不应只交给公司单方。
- 支付、短信、AI token、邮箱等密钥通过环境变量管理，不提交到 Git。
- 公司支付主体接入时，只改运营配置，不改权属说明。
- 每次上线记录 Git 提交哈希、部署时间、部署人和回归结果。

## 合作终止时的最低目标

合作无法继续时，个人至少应能做到：

1. 拿得出完整 Git 历史和源码证据。
2. 拿得出项目早期形成时间、作者、源代码范围和证据哈希。
3. 证明公司使用的是个人早期成果的延续版本。
4. 要求继续使用方补授权费、分成、买断款或停止使用。
5. 个人可另行继续开发、部署或申请软著/商标。

## 交接前必须执行

```bash
bash scripts/create_ip_evidence_package.sh
git bundle create artifacts/ip-evidence/$(date +%Y%m%d-%H%M%S)-xuan-cet-tai-git-history.bundle --all
git log --reverse --format='%h %H %ad %an %ae %s' --date=iso-strict > artifacts/ip-evidence/$(date +%Y%m%d-%H%M%S)-git-log.txt
```

导出后把 `artifacts/ip-evidence/` 同步到个人可控位置，不要只放在公司电脑或公司网盘。

## 不要做的事

- 不要写隐藏后门、停服开关、删除脚本或破坏性逻辑。
- 不要用代码故意限制公司正常备案、支付和运营。
- 不要把争议性私有说明提交到公司公开协作仓库。
- 不要口头同意“先都放公司名下以后再说”。
- 不要在未留证据前迁移仓库 owner、域名 owner 或生产服务器最高权限。
