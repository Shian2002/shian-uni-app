# 备案、支付与运营主体变更说明

本文档用于说明项目在备案主体、支付主体或运营账号发生变更时的工程边界和材料口径。它不展示给终端用户，也不参与前端构建。

## 基本原则

1. ICP 备案主体、微信支付商户主体、客服主体和开票主体属于运营主体安排。
2. 运营主体变更不等同于软件著作权、商标权、作品著作权或源代码所有权变更。
3. 若后续需要将知识产权转让、独占授权或以公司名义申请软著/商标，应另行形成书面文件。
4. 对外材料应区分“运营主体”和“权利主体”，避免使用会造成权属混同的表述。

## 推荐对外表述

可以使用：

- “公司负责网站备案、支付结算、客服和日常运营。”
- “公司为网站主办单位和支付服务主体。”
- “软件由项目发起人/技术负责人主导开发，公司用于项目运营。”

避免使用：

- “公司拥有本软件全部知识产权。”
- “公司完全自主研发并独占本项目软件。”
- “个人仅提供技术协助。”

## 代码与配置边界

运营主体变更时，通常只应涉及以下配置：

- ICP 备案号、备案主体展示信息。
- 微信支付商户号、回调证书、API key、结算主体配置。
- 公众号/小程序 AppID、客服、短信、邮件、企业认证资料。
- 发票、客服、隐私政策和用户协议中的运营主体信息。

不应因为运营主体变更而改动以下内容的权属口径：

- 源代码著作权归属。
- 品牌、Logo、产品名或商标申请主体。
- 数据库结构和核心业务逻辑的作者归属。
- 历史 Git 提交、源程序鉴别材料和软著申请材料。

## 变更前检查清单

- 已备份当前代码、配置和部署包。
- 已记录当前 Git 提交哈希和线上版本。
- 已确认备案/支付变更只影响运营信息，不改写知识产权归属表述。
- 已确认生产密钥不会直接提交到仓库。
- 已确认用户协议、隐私政策和支付页面的主体信息一致。
- 已确认测试环境和生产环境支付配置分离。

## 变更后检查清单

- ICP 信息展示正确。
- 支付下单、回调、退款和积分到账链路正常。
- 用户协议、隐私政策、客服信息和支付主体一致。
- 后端日志不泄露支付密钥、证书、身份证明材料或后台账号信息。
- Git diff 中没有出现公司或个人的敏感证件号码、密钥、支付证书内容。

## 建议补充文件

如果公司开始承担备案、支付和运营，应至少保留一份内部确认文件，说明：

- 项目早期版本和源代码由项目发起人/技术负责人主导形成。
- 公司作为运营主体使用该项目，用于备案、支付、客服、推广和开票。
- 项目未形成稳定收入前，不设置固定授权费。
- 项目产生稳定收入、融资、转让、独占合作或引入新股东前，另行签署正式协议。
